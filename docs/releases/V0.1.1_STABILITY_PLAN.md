# v0.1.1 Stability Plan

## Positioning

v0.1.1 is the engineering-stability line after the immutable v0.1.0 Release. It does not change
recipient protocol conclusions, does not change `live_verified=false`, and does not close Issue #1.

## Phase 1 — diagnostics and recovery foundation

Completed in PR #22:

- redacted runtime diagnostics;
- SQLite integrity, foreign-key, migration-name and checksum validation;
- production backup → isolated restore verification;
- Linux and Windows verification;
- explicit SQLite handle closure on Windows.

## Phase 2 — database maintenance and restart stability

Completed in PR #24:

- default read-only maintenance plans;
- explicit `--apply --confirm-stopped` mode;
- verified full runtime backup before writes;
- WAL checkpoint and `PRAGMA optimize` only;
- VACUUM recommendation without automatic execution;
- database-busy and migration-corruption fault injection;
- repeated recording and postprocess startup recovery with idempotency checks.

## Phase 3 — operator experience and release preparation

Implemented under Issue #25 and PR #26:

- source and portable `operations.bat` dispatchers;
- standalone redacted diagnostics entrypoints;
- maintenance-plan and explicitly confirmed maintenance-apply entrypoints;
- portable clean-extraction execution of diagnostics and maintenance-plan;
- Unicode-path Windows batch compatibility;
- diagnostics support for configured runtime database paths;
- README and operator documentation;
- v0.1.1 implementation report and Release notes draft.

Phase 3 does not update `pyproject.toml`, `app.__version__`, `release-lock.json`, tag, or Release.
Actual v0.1.1 publication remains a separate audited decision after Phase 3 is merged.

## Excluded

- real recipient protocol integration or claims;
- protobuf contract guesses;
- Redis, Celery, PostgreSQL, multi-machine architecture;
- public management access;
- OCR, face, voice, gift, comment, nickname, or screen-position fallback.

## Acceptance

Every implementation batch requires a feature branch, commit, push, Draft PR, exact-head CI,
Windows verification, source recovery, package verification, and merge to `main` only when green.
