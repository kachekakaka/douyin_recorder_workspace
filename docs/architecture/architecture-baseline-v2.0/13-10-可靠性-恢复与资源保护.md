# 10. 可靠性、恢复与资源保护

## 10.1 故障矩阵

| **故障**            | **媒体处理**                | **recipient 处理**        | **恢复动作**                           |
|---------------------|-----------------------------|---------------------------|----------------------------------------|
| IM 断线，媒体正常   | 继续录制                    | 关闭 Active，进入 Unknown | 独立退避重连，等待下一条有效事件       |
| 媒体断线，IM 正常   | 关闭 writing 文件，记录 gap | 保持当前状态机            | 重新解析流并创建新 continuity_group    |
| 状态接口失败        | 不立即停止                  | 不改变                    | 退避并告警                             |
| FFmpeg 退出且仍在线 | 记录退出码/gap              | 不改变                    | 达到阈值前重启；超过后 media_failed    |
| SQLite 短时繁忙     | 媒体继续                    | 有界事件缓冲              | DbWriter 重试；满载写 JSONL 灾备并告警 |
| 磁盘 soft limit     | 继续当前录制                | 不改变                    | 暂停后处理和新导出                     |
| 磁盘 hard limit     | 优雅保护当前文件            | 关闭场次                  | 停止启动新录制并高优先级告警           |
| 主服务重启          | 扫描 writing/残留进程       | 从 Unknown 恢复           | 重新 probe；不继承旧 recipient         |

## 10.2 重启恢复顺序

> **1.** 获取单实例锁，避免重复启动两个主服务。
>
> **2.** 验证 config、SQLite schema、FFmpeg/ffprobe 和目录权限。
>
> **3.** 扫描数据库中 active session、writing media_file、running job 和 sidecar。
>
> **4.** 结束遗留 FFmpeg 进程或按明确 PID/sidecar 归属接管；不允许无归属进程继续录制。
>
> **5.** 可 ffprobe 的 writing 文件标记 recovered/truncated；不可识别文件进入 quarantine。
>
> **6.** 重新查询房间：已下播则闭合场次并入后处理；仍直播则新建媒体文件和 IM，recipient 从 Unknown 开始。
>
> **7.** running job 改为 queued/retry，attempts + 1；超过上限进入 failed。

## 10.3 容量与优先级

| **优先级** | **任务**                    | **策略**                                       |
|------------|-----------------------------|------------------------------------------------|
| P0         | 在线媒体写入                | 不可被精确转码抢占；磁盘不足时优先关闭完整文件 |
| P1         | IM 接收和时间线写入         | 低数据量；短时缓冲；不能无限占内存             |
| P2         | 状态检查与 API              | 有界并发、超时、抖动                           |
| P3         | ffprobe、索引、fast-copy    | 有界队列，默认 1 个                            |
| P4         | precise、代理转码、批量清理 | 低峰或人工触发，可暂停                         |

单路每小时容量(GB)≈码率(Mbps)×0.45。上线前必须在目标硬件上做同时在线路数、总码率、磁盘写入和后处理组合压测，并据此设置 max_active_recordings。
