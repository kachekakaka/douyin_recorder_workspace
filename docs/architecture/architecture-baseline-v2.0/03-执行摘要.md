<th><p><strong>最终推荐</strong></p>
<p>构建一个单体 Python 服务：一个 RoomManager 管理多个 RoomWorker；每个在线房间独立运行状态检查、FFmpeg 连续录制、抖音 IM WebSocket 和推荐收礼人状态机。推荐对象变化只闭合旧区间并开启新区间，不重启媒体流；直播结束后再按时间线导出。管理界面使用静态网页，由 FastAPI 同源托管。</p></th>

本版刻意保持轻量：不设计多平台插件体系，不引入 PostgreSQL、Redis、Celery、消息总线或多机租约。首版以单机、单进程、一个 Uvicorn worker、SQLite WAL 和外部 FFmpeg 为明确边界。只有当真实压测证明单机不足时，再另立升级方案。

| **保留**                               | **删除或延后**          | **原因**                                    |
|----------------------------------------|-------------------------|---------------------------------------------|
| 多直播间、每房间独立 FFmpeg 与 IM      | 多平台适配器/能力系统   | 当前只使用抖音，避免抽象成本                |
| 严格推荐收礼人状态机                   | Rust/Tokio              | Python asyncio 足以承担控制面与少量事件处理 |
| 原始媒体连续录制、直播后切片           | PostgreSQL/Redis/Celery | 单机 SQLite + 持久化任务表更易部署和恢复    |
| SSE、网页管理、Range 播放              | 前端框架与构建链        | 静态 ES Modules 可直接运行、发布更简单      |
| GitHub CI、标签、Git Bundle 与数据备份 | 仅依赖 GitHub           | 源码与运行数据必须双轨备份                  |

- 媒体链路与身份事件解耦：媒体尽量连续，消息链路只负责形成可审计时间线。

- 首事件前、空 recipient、IM 断线期间一律为 Unknown；不继承断线前成员。

- 配置、运行状态和原始媒体分目录；Git 只保存源码、模板、测试、文档与脚本。

- 每次重要发布同时保留 tag、源码 ZIP、完整 Git Bundle、SHA-256 和验证报告；GitHub 不是唯一备份。
