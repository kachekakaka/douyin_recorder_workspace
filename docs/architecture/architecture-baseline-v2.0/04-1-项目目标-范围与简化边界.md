# 1. 项目目标、范围与简化边界

## 1.1 项目目标

建设一套面向抖音团播的长期监听与录播工作台：用户可以在网页中配置多个直播间，系统自动检测开播并并发保存原始直播流，同时根据抖音团播的“推荐收礼人变化”形成成员时间线。直播结束后，用户可查看完整录像、Unknown 区间和按成员/轮次生成的导出任务。

## 1.2 必选需求

| **编号** | **需求**               | **验收口径**                                                      |
|----------|------------------------|-------------------------------------------------------------------|
| R-01     | 多个抖音直播间常驻监听 | 网页可增删、启用、停用；离线按间隔检查                            |
| R-02     | 多个直播间同时录制     | 每个在线房间独立 FFmpeg 子进程，单房故障不影响其他房间            |
| R-03     | 唯一身份信号           | 只处理 WebcastGroupLiveGiftRecipientRecommendMessage              |
| R-04     | 严格 Unknown           | 首事件前、空 recipient、IM 断线均不得继承或猜测                   |
| R-05     | 对象切换不重启拉流     | FFmpeg PID 与媒体连接不因成员切换而变化                           |
| R-06     | 直播后按时间线导出     | 保留完整原始媒体；fast-copy 默认，precise 按需                    |
| R-07     | 后端纯 Python          | FastAPI/asyncio/httpx/websockets/protobuf；媒体由外部 FFmpeg 处理 |
| R-08     | 前端纯网页             | 静态 HTML/CSS/原生 JavaScript ES Modules，无 npm 构建依赖         |
| R-09     | 可重启恢复             | 活动场次、writing 文件、未完成任务按明确规则恢复                  |
| R-10     | 可运维                 | 状态 API、SSE、结构化日志、磁盘阈值、诊断页面                     |
| R-11     | 可直接运行             | Windows start.bat；Docker Compose；verify.bat 可自检              |
| R-12     | 源码和数据防丢         | GitHub、tag、Git Bundle、独立运行数据备份与恢复演练               |

## 1.3 明确不做

- 不支持 Bilibili、快手或其他平台；不为未来平台设计插件市场。

- 不使用 OCR、人脸、声纹、礼物接收人、弹幕、连麦成员或画面位置推断当前对象。

- 不在首版提供多机 Worker、房间租约、Redis、NATS、Kafka 或 Kubernetes。

- 不在直播过程中按成员切断媒体连接；不自动对全部录像做精确转码。

- 不做自动投稿、内容识别、剪辑包装、商业分析或开放注册。
