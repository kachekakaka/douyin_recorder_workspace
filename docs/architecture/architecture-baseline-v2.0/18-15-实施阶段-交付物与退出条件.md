# 15. 实施阶段、交付物与退出条件

| **阶段**          | **目标**                      | **主要交付物**                                                                         | **退出条件**                                                             |
|-------------------|-------------------------------|----------------------------------------------------------------------------------------|--------------------------------------------------------------------------|
| P0 仓库与协议验证 | 可恢复工程骨架 + 验证目标消息 | GitHub 私有仓库、CI、start/update/verify/backup；单房间 method dump、fixture、字段样本 | Git Bundle 可恢复；目标房间切换时稳定收到 user_id/open_id；解释空值/重连 |
| P1A 单房间媒体基础 | 房间与连续媒体基础           | 房间 CRUD、直播页/流候选解析、RecorderSupervisor、schema v2、基础网页/API | 合成 fixture 不泄密；本地媒体 smoke 通过；不依赖 recipient 现场结论      |
| P1B 单房间完整闭环 | 连续录制 + 真实状态机接线     | StreamResolver、场次/media 入库、真实 IM、RecipientStateMachine、恢复 | 单场完整录制；对象切换不重启媒体；IM 断线产生 Unknown；协议事实可审计   |
| P2 多房间与网页   | 多个房间长期运行              | RoomManager/RoomWorker、动态房间管理、SSE、概览/房间/实时页面                          | 多房间同时开播互不影响；单房故障隔离                                     |
| P3 后处理与播放   | 按区间查看和导出              | JobRunner、index-only、fast-copy、Range、MP4 proxy、任务页                             | 任一导出可追溯到 interval 和 media_files；重启后任务恢复                 |
| P4 生产化与防丢   | 可长期运行和恢复              | 认证、审计、磁盘保护、崩溃恢复、Windows runtime、Docker、发布资产                      | 长时间压测通过；从 Git Bundle + 数据备份完整恢复                         |

## 15.1 P0 必须先验证

> **1.** 进入直播间后是否立即收到当前推荐收礼人，还是只有变化时才下发。
>
> **2.** 推荐对象切换时是否每次都下发，是否存在重复帧。
>
> **3.** 无推荐对象时 user_id/open_id 的实际值。
>
> **4.** 重连后是否重放当前对象。
>
> **5.** change_reason_enum 的实际取值分布；只记录，不作为判断条件。
>
> **6.** common.create_time 与本地 received_at 的延迟分布。
>
> **7.** 同一房间/不同房间的 Cookie、ttwid、签名和流地址失效行为。


## 15.2 P1 拆分说明

P0 的仓库与合成协议工具可以完成，而真实目标房间的 IM 现场事实可能受开播时间、游客上下文和风控影响。为避免把协议不确定性阻塞在所有媒体工程之前，P1 拆成 P1A/P1B：P1A 只推进房间、流候选和 FFmpeg Supervisor，并保持 `live_verified=false`；P1B 才把经审查的真实目标消息接入 session/interval。P1A 通过不得被描述为完整 P1 已验收。
